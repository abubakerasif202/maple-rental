# Digital Lease Agreement Management Design

## Purpose
To provide a persistent storage system for car lease agreements. This allows the administrator to save generated leases, maintain historical records, and easily retrieve them for reference or auditing purposes.

## Understanding Summary
- **What is being built:** A "Persistence Layer" for car lease agreements.
- **Why it exists:** To transition from a temporary, "live" agreement generator to a permanent record-keeping system.
- **Who it is for:** The sole administrator of the platform.
- **Key constraints:**
  - **New Database Table:** `lease_agreements` stores rendered markdown and relationships.
  - **Dashboard Integration:** Added "Save Agreement" and "Agreement History" functionality.
  - **Full-Text Persistence:** Storing the entire rendered text ensures historical accuracy.
- **Explicit non-goals:** Digital signatures (e.g., DocuSign), PDF generation, or a public-facing portal for drivers.

## Assumptions
- Markdown remains the primary format for the agreements.
- The administrator manually clicks "Save Agreement" when satisfied with the generated text.
- The `lease_agreements` table will reference existing `applications` and `cars` tables.

## Decision Log
| Decision | Alternatives Considered | Reason for Choice |
| :--- | :--- | :--- |
| **Full-Text History (Approach A)** | JSON Data, File Storage | Ensures absolute historical reliability. Even if the template or driver data changes later, the saved lease remains exactly as it was when generated. |
| **Dashboard History View** | No History, Separate Page | Keeps the workflow unified within the "Lease Agreement" tab for the administrator. |
| **Relational Linking** | Plain Text Only | Linking to `application_id` and `car_id` allows for fast filtering and reporting in the future. |

## Final Design

### 1. Database Schema (`lease_agreements` table)
```sql
CREATE TABLE lease_agreements (
  "id" BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  "applicationId" BIGINT NOT NULL REFERENCES applications("id") ON DELETE CASCADE,
  "carId" BIGINT NOT NULL REFERENCES cars("id") ON DELETE CASCADE,
  "content" TEXT NOT NULL,
  "status" TEXT DEFAULT 'generated' NOT NULL,
  "createdAt" TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP NOT NULL
);
CREATE INDEX idx_lease_agreements_application ON lease_agreements("applicationId");
CREATE INDEX idx_lease_agreements_car ON lease_agreements("carId");
```

### 2. API Endpoints
- **POST `/api/agreements`:** Saves the full rendered Markdown text along with `applicationId` and `carId`.
- **GET `/api/agreements`:** Fetches a list of all saved agreements with basic metadata.
- **GET `/api/agreements/:id`:** Retrieves the full content of a specific agreement.
- **DELETE `/api/agreements/:id`:** Allows the admin to remove an erroneous or outdated record.

### 3. Dashboard Integration (`AdminDashboard.tsx`)
- **"Save Agreement" Button:** A new button next to "Download" that sends the `generatedAgreement` text to the API.
- **"History" Section:** A small table or list below the builder showing previously saved agreements (Driver Name, Date, Car, and a "View" button).
- **"View" Modal:** A simple modal to display the historical Markdown text of a saved agreement.
