# Deploy Maple Rental to Render

## Architecture

This repository is configured for a single Render Web Service.

That one service does all of the following:
- Runs the Express backend API
- Serves the built Vite frontend from `dist/`
- Handles SPA client-side routes by falling back to `dist/index.html`

## Why this setup was needed

Before these changes, the project was not ready for a clean single-service Render deployment because:
- `npm start` used `vite-node`, which is a development runtime rather than a production server entrypoint
- the backend served `dist/` only when `NODE_ENV=production`, but there was no compiled production backend build for Render to start
- static frontend serving and SPA fallback were tied to the development-oriented startup flow
- there was no health endpoint for Render to probe
- missing Supabase env vars only produced a warning and created a dummy client, which is unsafe in production

## Render Blueprint

This repo includes `render.yaml`, so Render can create the service from source control.

Blueprint link:
[https://dashboard.render.com/blueprint/new?repo=https://github.com/abubakerasif202/maple-rental](https://dashboard.render.com/blueprint/new?repo=https://github.com/abubakerasif202/maple-rental)

## Render Service Settings

- Service type: `Web Service`
- Runtime: `Node`
- Build command: `npm ci && npm run build`
- Start command: `npm start`
- Health check path: `/api/health`

## Production runtime behavior

### Backend entrypoint

The production backend entrypoint is:
- `server-dist/api/index.js`

It is generated by:
- `npm run build:server`

### Frontend build system

The frontend is built with:
- Vite

The frontend production build output is:
- `dist/`

### Production start flow

`npm start` runs the compiled Express backend:

```bash
npm start
```

The backend then:
- listens on `0.0.0.0` and `process.env.PORT`
- serves static files from `dist/`
- serves `dist/index.html` for non-API SPA routes
- keeps `/api/*` handled by Express routes

## Environment variables

### Required

- `APP_URL`
  - Purpose: canonical public URL for redirect and return links
  - Used in: `api/index.ts`, `api/routes/saas.ts`
  - Render value: your Render service URL, for example `https://your-service.onrender.com`

- `ADMIN_EMAIL`
  - Purpose: single allowed admin login email in production
  - Used in: `api/routes/auth.ts`, `api/routes/applications.ts`

- `SUPABASE_URL`
  - Purpose: backend Supabase project URL
  - Used in: `api/db/index.ts`

- `SUPABASE_SERVICE_ROLE_KEY`
  - Purpose: privileged backend Supabase key for server-side database operations
  - Used in: `api/db/index.ts`

- `STRIPE_SECRET_KEY`
  - Purpose: Stripe server SDK key used by payment, webhook, SaaS, and financial routes
  - Used in: `api/routes/stripe.ts`, `api/routes/webhooks.ts`, `api/routes/saas.ts`, `api/routes/financials.ts`

- `VITE_STRIPE_PUBLIC_KEY`
  - Purpose: Stripe publishable key injected into the frontend build
  - Used in: `src/pages/Apply.tsx`, `src/pages/Checkout.tsx`
  - Note: this must be present at build time on Render because Vite inlines it into the frontend bundle

### Optional

- `STRIPE_WEBHOOK_SECRET`
  - Purpose: verify Stripe webhook signatures
  - Used in: `api/routes/stripe.ts`, `api/routes/webhooks.ts`
  - Required if Stripe webhooks are enabled

- `RESEND_API_KEY`
  - Purpose: send transactional emails
  - Used in: `api/routes/applications.ts`, `api/routes/stripe.ts`, `api/routes/webhooks.ts`

- `FRONTEND_URL`
  - Purpose: optional legacy CORS origin override
  - Used in: `api/index.ts`, `api/routes/saas.ts`
  - Not needed for the single-service Render deployment if `APP_URL` is set correctly

- `CORS_ORIGIN`
  - Purpose: optional extra allowed browser origin
  - Used in: `api/index.ts`

### Platform-provided

- `PORT`
  - Purpose: port assigned by Render
  - Used in: `api/index.ts`
  - Do not set this manually on Render

- `NODE_ENV`
  - Purpose: enables production behavior
  - Used in: `api/index.ts`, `api/db/index.ts`, `api/routes/auth.ts`
  - Set by `render.yaml` to `production`

## Local development

Local development remains aligned with the previous workflow:

```bash
npm ci
npm run dev
```

That continues to run the Express API with Vite middleware in development mode.

## Local production verification

Use these commands to simulate Render locally:

```bash
npm ci
npm run build
$env:NODE_ENV='production'
$env:PORT='3000'
npm start
```

Then verify:
- `http://localhost:3000/api/health`
- `http://localhost:3000/`
- `http://localhost:3000/admin/dashboard`

## Notes

- The backend now fails fast in production if `SUPABASE_URL` or `SUPABASE_SERVICE_ROLE_KEY` is missing.
- Static assets are served directly from `dist/`.
- SPA fallback applies only to non-API routes, so missing API endpoints do not return the frontend shell.
